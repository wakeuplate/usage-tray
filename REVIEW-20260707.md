# Limit Lens PLAN.md Review — 2026-07-07

Reviewer: Claude (Haiku 4.5, acting as review-only; no code changes)

## Verdict: Largely Acceptable — Needs Revisions Before Implementation

Overall architecture is sound, security design is mature (sessionKey risk analysis is better than most side projects), and the UI spec is concrete enough for implementation. However, several key technical feasibility assumptions are unverified, and must be validated before development begins.

---

## Issues (sorted by severity)

### P1 — Claude safe-mode data source likely cannot provide 5h usage percentage [Critical]

**Problem:** PLAN assumes `statusLine hook` or `ccusage` can provide 5h/weekly usage percentage and reset time. In practice, `ccusage` only provides token counts and cost from local logs. The actual quota percentage and reset countdown are server-side information with no official local API. `statusLine` outputs custom command stdout (and doesn't work in VS Code extension).

**Impact:** Claude's half of the UI (percentage bar + countdown) may be impossible to render in safe mode — this conflicts with the core UI design.

**Recommendation:** Before any implementation, verify exactly which fields are available from `ccusage`, `~/.claude/usage.db`, and related local sources. If percentage/reset cannot be obtained, downgrade Claude safe-mode UI to "token consumption trend only" (no percentage bar, no countdown). Reserve percentage bar for sessionKey precision mode. Consider whether this changes the project's value proposition enough to warrant promoting sessionKey to v1.

### P2 — Codex data sources are listed but none verified [Critical]

**Problem:** Three Codex sources listed (`codex app-server`, `sessions/*.jsonl`, `logs_2.sqlite`) but no actual field names, schema dumps, or confirmation that any of them contain rate limit / quota information. Codex quota management is server-side; local logs may only have token counts.

**Impact:** Same as P1 — Codex percentage bar and countdown may also be unrenderable.

**Recommendation:** Run verification before development: `ls -la ~/.codex/`, dump JSONL sample fields, `sqlite3 ~/.codex/logs_2.sqlite .schema`. This can be delegated to Codex itself.

### P3 — "5h usage rate" definition is undefined [High]

**Problem:** Anthropic has never published the exact 5h window calculation (sliding vs fixed, token-based vs request-based, model weighting). The PLAN's bar charts, percentages, and reset countdowns all assume this number is known and obtainable.

**Impact:** Even with sessionKey, the percentage may be Anthropic's internal calculation that changes without notice.

**Recommendation:** Explicitly acknowledge in the PLAN that all usage percentages are estimates. UI should label them as "estimated" when derived from local data. Reset time can be approximated as "5h from earliest usage record in window" but must be marked as estimated.

### P4 — Tauri v2 tray popup positioning on Windows multi-monitor + DPI scaling [Medium]

**Problem:** The desired behavior (380x460 popup anchored to tray icon) is non-trivial on Windows with mixed DPI monitors. PLAN acknowledges the risk but provides no mitigation.

**Recommendation:** Add an early spike task: minimal tray icon + popup window positioning test at 125%/150% scaling with multi-monitor. Do this before investing in UI implementation. Suitable for Codex execution.

### P5 — `tokens_total` and `estimated_cost` sources are ambiguous [Medium]

**Problem:** `usage_snapshots` table records these fields but doesn't specify which data source provides them for each agent, or how cost is calculated (rate table maintenance).

**Recommendation:** Add a data source mapping: "Claude token/cost = ccusage usage.db values", "Codex token/cost = parsed from local JSONL/SQLite". Specify where the pricing rate table lives and how it's updated.

### P6 — `limit_cycles` table depends on reset times that may be unavailable [Medium]

**Problem:** `cycle_start_at` / `cycle_reset_at` require knowing exact reset boundaries, which may not be obtainable (see P1, P3).

**Recommendation:** If reset times are estimated, cycle table should be marked as approximate. Alternative: track usage trends via continuous snapshots without trying to precisely segment cycles.

### P7 — Mobile dashboard lacks authentication [Low-Medium]

**Problem:** v1.5 local web dashboard on LAN IP has no authentication. Anyone on the same network can view usage data.

**Recommendation:** Add minimum authentication: display a PIN on desktop, require it on first mobile connection. Simple but blocks casual snooping.

### P8 — No auto-start design [Low]

**Problem:** Windows tray tools typically need "start with Windows" capability. Not mentioned in PLAN.

**Recommendation:** Add "auto-start toggle in Settings" using Tauri's autostart plugin (Registry Run key).

### P9 — No update mechanism [Low]

**Problem:** Codex/Claude log formats may change with updates. No mention of version checking or auto-update.

**Recommendation:** v1 minimum: check GitHub releases on startup, show notification if newer version available. No auto-install needed.

### P10 — Settings page content not specified [Low]

**Problem:** Popup has a Settings icon but no description of what settings are available.

**Recommendation:** Add Settings page field list: data source paths, refresh interval, theme, auto-start, history retention days, sessionKey precision mode toggle (v1.2).
