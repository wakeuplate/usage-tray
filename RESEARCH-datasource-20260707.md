# Research: Claude/Codex Usage Data Sources

`sr-kai/claudeusagewin` reads Claude Code's local OAuth credential file, not browser cookies or statusLine. `visualstudio-project/ClaudeUsage/ClaudeUsage/Services/CredentialService.cs:28-39` builds the Windows path as `%USERPROFILE%\.claude\.credentials.json`; `CredentialService.cs:81-105` also scans WSL homes at `\\wsl.localhost\<distro>\home\<user>\.claude\.credentials.json` and `\\wsl$\<distro>\home\<user>\.claude\.credentials.json`, then `CredentialService.cs:137-145` chooses the most recently modified candidate. `CredentialService.cs:155-195` reads the JSON, deserializes `ClaudeAiOauth`, checks `ExpiresAt`, refreshes when the access token expires in under five minutes, and returns `AccessToken`; `CredentialService.cs:232-247` shows the refresh response fields it keeps are `access_token`, `refresh_token`, `expires_in`, and `scope`. It does not appear to extract an org id from `.credentials.json`; the usage request is a single OAuth usage endpoint call. `visualstudio-project/ClaudeUsage/ClaudeUsage/Services/UsageApiService.cs:17` defines `https://api.anthropic.com/api/oauth/usage`, and `UsageApiService.cs:173-193` sends `Authorization: Bearer <token>`, `User-Agent: claude-code/<version>`, and `anthropic-beta: oauth-2025-04-20`, then deserializes the JSON into `UsageData`. The README confirms the same behavior and says the app reads `~/.claude/.credentials.json`, sends required `claude-code/*` and `anthropic-beta` headers, and uses an undocumented API (`README.md:269-302` on the GitHub page). The endpoint therefore looks like Claude Code's internal OAuth usage API, not the public Anthropic Messages/Admin API.

For `sr-kai/claudeusagewin`, the `.credentials.json` approach appears to return the data limit-lens needs: the app advertises session and weekly gauges plus reset countdowns, and its service parses the OAuth usage response directly into UI data (`UsageApiService.cs:173-193`; README `Features` lines 244-257 and `How It Works` lines 292-302 on GitHub). It also has explicit token refresh behavior: `CredentialService.cs:173-195` refreshes before expiry and `CredentialService.cs:198-229` posts `grant_type=refresh_token` plus `refresh_token` to `https://console.anthropic.com/v1/oauth/token`, then writes the refreshed `ClaudeAiOauth` back to the same credential file at `CredentialService.cs:225-229`. The risk is not read-only in a formal permission sense: the access token is a bearer credential for Claude Code OAuth, and the repo does not demonstrate scoped-down usage-only permissions. The `scope` string is merely stored from refresh responses (`CredentialService.cs:232-247`), not enforced locally. Compared with QuotaGem's `sessionKey` cookie approach, this is probably a smaller incremental attack surface because it reuses the exact secret Claude Code already stores after login, whereas `sessionKey` comes from the web session cookie jar; however, any process that reads this token can call at least the undocumented usage endpoint and likely whatever other Claude Code OAuth endpoints accept the same bearer token.

`Finesssee/Win-CodexBar` does not mean "no credentials"; it means the app can reuse existing local/browser credentials instead of prompting its own login. Its README says first run requires adding/selecting a credential type, and for Claude specifically "browser cookies/sessionKey are preferred" while OAuth and CLI are fallbacks (`README.md:314-321` on GitHub). The Claude web provider reads a session key from `CLAUDE_AI_SESSION_KEY` or `CLAUDE_WEB_SESSION_KEY` (`rust/src/providers/claude/web_api.rs:2376-2386` and `web_api.rs:2668-2703`) or imports browser cookies from `claude.ai`, `claude.com`, `console.anthropic.com`, and `anthropic.com` (`web_api.rs:2389-2429`). It then calls the Claude web API at `https://claude.ai/api`: `web_api.rs:2355-2357` defines the base URL; `web_api.rs:2706-2768` obtains the org id from `lastActiveOrg`, account membership, or `/organizations`; `web_api.rs:2770-2808` calls `/organizations/{org_id}/usage`; and `web_api.rs:2811-2856` optionally calls `/organizations/{org_id}/overage_spend_limit`. For Codex, it uses the Codex CLI's local auth file: `rust/src/providers/codex/mod.rs:528-530` says OAuth credentials are stored by Codex CLI in `~/.codex/auth.json`; `rust/src/providers/codex/api.rs:2240-2260` defines `https://chatgpt.com/backend-api` plus `/wham/usage`; `api.rs:2306-2335` sends `Authorization: Bearer <access_token>` and optional `ChatGPT-Account-Id`; `api.rs:2459-2495` reads the auth file; `api.rs:2499-2546` accepts either `OPENAI_API_KEY` or `tokens.access_token`; and `api.rs:2627-2644` resolves `$CODEX_HOME/auth.json` first, otherwise `~/.codex/auth.json`.

Bonus: `nek0der/CodexBarWin` appears to be a Windows UI wrapper around an external CodexBar CLI, not an independent Claude datasource. Its README states the app is a "WinUI 3 frontend wrapper for CodexBar CLI" (`README.md:359-364` on GitHub), requires WSL and the CodexBar CLI (`README.md:384-390`), and tells users to verify refresh with `wsl codexbar --provider claude --format json` (`README.md:471-475`). So its Claude datasource is whatever CodexBar CLI uses, not statusLine or `.credentials.json` in this repo.

Bonus: `babakarto/CodexBar-Win` could only be checked at the repository/README level in this pass because shell network cloning was blocked and the GitHub connector call was cancelled. The visible README describes it as a Windows system tray app for Claude Code usage monitoring with session limits, weekly limits, reset times, and API costs (`README.md` repository title/description on GitHub), but the browsed repository page did not expose enough source detail to classify its datasource as statusLine, `.credentials.json`, or `sessionKey`. Treat this one as unresolved until source files can be fetched through a working GitHub API/clone path.

Recommendation: replace the statusLine collector with a Claude Code OAuth `.credentials.json` collector as the primary path, but keep it behind a clearly named "uses Claude Code local OAuth token" permission boundary. It solves the two statusLine defects because it can query on demand without a live CLI session and appears to expose 5h/weekly usage plus reset metadata through `https://api.anthropic.com/api/oauth/usage`. It is likely lower incremental risk than `sessionKey` because it uses the credential Claude Code already depends on, but it is not inherently read-only and relies on an undocumented internal API; the safest implementation is read-only credential access where possible, no token logging, no copying the token into app config, explicit refresh behavior only if necessary, and a cookie/sessionKey fallback only as an opt-in last resort.
